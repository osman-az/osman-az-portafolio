Consultor de Users de la API-REST reqres.in

Frontend Web que usa la API-Rest del sitio web reqres.in. Para consultar users y modificarlos.

  • Tech: Bootstrap, Javascript, HTML, CSS, jQuery, SASS, SweetAlert, Fetch

Consultor de Users de la API-REST reqres.in

Ver en GitHub Ver Online

Descripción Detallada

Usando la API-Rest del sitio web https://reqres.in/ que contiene la documentación de un API pública. He creado un archivo HTML nuevo y un archivo JS para programar las instrucciones necesarias para cumplir con los siguientes requerimientos, utilizando Bootstrap para la maquetación, fetch para hacer las peticiones al backend, jQuery para manipular el HTML y SweetAlert para mostrar avisos al usuario.

Características:

  1. Al cargar la página (ready), mostrar el listado de todos los "users" (Read de todos los recursos, es decir, un get de todos los recursos). Mostrar una imagen loading mientras llega la respuesta y luego esconder la imagen de loading cuando se muestren los resultados.

  2. Botón con el texto "Agregar" que muestra una ventana modal con inputs necesarios para introducir los datos de un nuevo "user". En la ventana modal esta el botón "Guardar" que hace un llamado al API para registrar el nuevo "user" (Create de un recurso, es decir, un post de un recurso). Cuando la respuesta es exitosa, se muestra un mensaje de éxito y esconder la ventana modal, en caso contrario, se muestra un mensaje de error y mantiene visible la ventana modal. Si el registro es exitoso, el nuevo registro debe aparecer al final del listado del objetivo anterior.

  3. Al hacer click sobre un "user", se muestra una ventana modal con inputs necesarios para modificar los datos del "user". En la ventana modal colocar un botón "Guardar" que haga un llamado al API para modificar el "user". Para lograrlo, debe hacer un llamado al API para obtener los datos del "user" sobre el cual se hizo click (Read de un recurso especifico, es decir, get de un recurso particular usando su identificador único). Al llegar la respuesta del backend, se debe mostrar en los inputs ventana modal inicializados con los datos recibidos del backend, para que el usuario pueda modificarlos.

  4. Al hacer click en el botón "Guardar", se hace un llamado al API para hacer update de los datos del usuario (Update de un recurso). Al llegar la respuesta del API, mostrar un mensaje de éxito o error dependiendo de la respuesta.

  5. Al lado de cada usuario, un botón que permita eliminar ese "user" (Delete de un recurso usando su identificador único). Está programado que al hacer click sobre ese botón, hacer un llamado al API para eliminar ese "user". Si la respuesta es exitosa, se debe eliminar ese usuario del listado (mostrar una imagen de loading mientras llega la respuesta).

  • Tech: Javascript, HTML, CSS, jQuery, Ajax, SASS, SweetAlert

CADIF1: Consultar Registros

Ver en GitHub Ver Online

Descripción Detallada

FrontEnd Web alternativo a la empresa CADI F1, usando los datos que se obtienen del API https://api.cadif1.com. Se creó un archivo index con un navbar con 4 opciones: Principal, Cursos, Carreras y Registro, y un div vacío. Se crearon 4 archivos HTML adicionales al index: uno con el nombre secciones.html para mostrar las secciones que están por comenzar, otro con el nombre cursos.html para mostrar los cursos que dicta la academia, otro con nombre registro.html y otro con el nombre carreras.html para mostrar las carreras.

Características:

  1. Usando el método load de jQuery, al hacer click en su respectiva opción (Principal -> secciones.html, Cursos -> cursos.html, Carreras -> carreras.html, Registro -> registro.html), se cargan en el div vacío del index, el HTML de cada página, mostrando inicialmente (es decir, en el ready del index) el contenido del archivo secciones.html. No hay hipervínculos tradicionales con el atributo href apuntando al archivo HTML porque se muestra el contenido HTML de cada página sin que cambie la URL en el navegador.

  2. Usando getJson de jQuery, se obtiene y muestra en cada página lo que retorna el ENDPOINT correspondiente: en secciones.html lo que retorna https://api.cadif1.com/seccion, en cursos.html lo que retorna https://api.cadif1.com/curso y carreras.html lo que retorna https://api.cadif1.com/carrera.

  3. Creado el archivo HTML registro.html que permite a un alumno registrarse como interesado a un curso. En este archivo agrega un formulario HTML para solicitar los datos personales del alumno (cédula, nombre y correo) y también tiene un SELECT con las áreas de estudio disponibles en la academia, en otro SELECT los cursos disponibles en la academia.

  4. Se validan que los datos del formulario no estén vacíos antes de enviarlos. Se muestran los mensajes de error de validaciones con SweetAlert. Si todo es válido, mostrar un confirm usando SweetAlert. Si el usuario responde que sí, usando el método $.post de jQuery se envían los datos al endpoint https://api.cadif1.com/test.

  5. Si el registro es exitoso, se programó que se muestre un mensaje de éxito y se limpie el formulario, pero si el registro no es exitoso, se muestra un mensaje de error y no se borra el formulario. Los mensajes de error se muestran usando SweetAlert.

Memory Game

Ver en GitHub Ver Online

Descripción Detallada

Para simular el famoso juego de cartas llamado MEMORIA, haciendo uso de la modificación del DOM usando solo Javascript.

Características:

  1. Botón con el texto "Iniciar partida", que averigüe la fecha y hora actual de la computadora para tomarla como momento de inicio del juego. Mostrar en la página la cantidad de tiempo que se tiene jugando. El texto del botón debe cambiar a "Reiniciar partida".

  2. Al momento de iniciar la partida, se deben repartir las cartas aleatoriamente, de modo que, en cada partida, las cartas no estén en el mismo lugar.

  3. Cuando un usuario hace click sobre una carta boca abajo y luego hace click sobre otra carta boca abajo, si la segunda carta es igual que la primera, se debe mostrar un mensaje de éxito y se debe incrementar un contador. Si el usuario vuelve a hacer click sobre una carta que pertenece a una pareja que ya ha sido encontrada, el programa no debe hacer nada. Si la segunda carta volteada no es igual que la primera, se debe mostrar un mensaje de error y ambas cartas deben volver a voltearse boca abajo.

  4. Cuando el usuario consiga todas las parejas, debe mostrar un mensaje con el texto "Ganaste" y detener el contador de tiempo. Se debe buscar en el local storage si hay algún tiempo almacenado, en caso de haber un tiempo almacenado, se debe comparar ese tiempo almacenado con el tiempo de juego actual. Si el tiempo de juego actual es menor al tiempo almacenado, se muestra un mensaje al usuario informando que batió el record y se debe actualizar el valor de tiempo almacenado con el valor de tiempo actual, en caso contrario, se mantiene el record.

  5. En cualquier momento que el usuario haga click en el botón "Reiniciar partida", se deben voltear todas las cartas, reiniciar el tiempo de juego y el contador de cartas encontradas, así como volver a colocar en el botón el texto "Iniciar partida".

Dos Formularios: Validación

Ver en GitHub Ver Online

Descripción Detallada

HTML que contiene 2 formularios. En un archivo Javascript aparte, están definidos 3 arreglos vacíos para almacenar los nombres de usuarios, los correos y las claves.

Características:

  1. En el formulario 2, se validan en el evento “onkeypress” que el usuario no pueda escribir espacios en blanco en el input "Nombre de usuario".

  2. En el evento “onblur” del input "Dirección de correo electrónico", usando expresiones regulares, se valida que el correo electrónico tenga el formato correcto.

  3. Se valida en el evento “onblur” del campo "Contraseña", que la clave tenga al menos 4 dígitos. En el evento “onkeyup” se valida que el campo "Confirmación de la contraseña", cuyo valor escrito en ese input sea igual al valor escrito en el input "Contraseña".

  4. En el evento "onclick" del botón "Registrarse" del formulario 2, se valida que ninguno de los inputs de ese formulario este vacío, que la clave y la confirmación de la clave sean iguales y que ni el correo ni el nombre de usuario existan en los arreglos. En caso de existir, se muestra un div con un mensaje de error, en caso contrario, se agregan los valores a los respectivos arreglos, se muestra un mensaje de confirmación de registro exitoso.

  5. En el formulario 1, se programó que cuando el usuario haga click en el botón "Acceder", se busque lo que el usuario escribió en el campo "nombre de usuario o correo" en los arreglos de nombres de usuarios o correos (puede ser uno u otro, no ambos a la vez) y lo que escribió en el campo "Contraseña" en el arreglo de claves. Cuando ambos valores son encontrados en los arreglos, se envía al usuario a una página de bienvenida (usando window.location), en caso de que no sean encontrados los datos, se muestra un div con un mensaje de error: "Usuario o clave no existen".
